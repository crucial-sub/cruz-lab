# LLM Yatcha: `llm-council`을 실제로 계속 써볼 만한 서비스로 다듬기

> 이 프로젝트는 Andrej Karpathy의 `llm-council`을 fork한 뒤, 직접 써보며 거슬렸던 부분을 손본 프로젝트다.  
> 핵심 토론 구조는 유지하되, 멀티 프로바이더 연동과 합의 가시화, Context 압축, 기본 UX를 보강해 실제 사용 흐름에 더 가깝게 다듬었다.

`LLM Yatcha`는 여러 모델이 같은 질문에 답하고, 서로의 답변을 익명으로 평가한 뒤, 마지막에 하나의 합의안으로 정리하는 실험 서비스다.

출발점이 된 구조는 분명히 있었다.  
대신 내가 집중한 건 "이걸 실제로 계속 써보려면 어디부터 손봐야 하나"였다.

- OpenRouter 단일 의존을 줄이고 싶었다
- 결과를 더 해석 가능하게 만들고 싶었다
- 긴 대화를 버틸 수 있게 만들고 싶었다
- 서비스처럼 쓰기 위한 기본 UX가 더 필요했다

## 출발점은 `llm-council`이었다

원본 `llm-council`의 3단계 구조는 꽤 매력적이었다.

특히 Stage 2의 익명 평가 방식이 좋았다.  
모델 이름이 아니라 `Response A`, `Response B`처럼 라벨을 붙여 서로 평가하게 만든다는 점이 단순 응답 비교보다 훨씬 흥미로웠다.

다만 막상 "내가 실제로 계속 쓸 만한 형태인가?"라고 생각하면 아쉬운 부분이 보였다.

가장 크게 걸렸던 건 두 가지였다.

1. **모델 연결 방식**  
   OpenRouter 중심 흐름은 간단하지만, 내가 이미 갖고 있는 API 키나 로컬 모델(Ollama)을 직접 쓰는 쪽과는 결이 달랐다.

2. **사용감**  
   결과는 나오지만, 합의가 얼마나 잘 됐는지 한눈에 읽기 어렵고, 재생성이나 대화 관리 같은 기본 UX도 더 손볼 여지가 있었다.

그래서 이 프로젝트의 방향은 처음부터 명확했다.  
**구조를 새로 발명하는 대신, 직접 써도 덜 거슬리게 만들자**는 쪽이었다.

## 가장 먼저 손본 건 멀티 프로바이더 직접 연동이었다

OpenAI, Anthropic, Google, xAI, Ollama는 겉으로 보면 다 비슷해 보인다.  
막상 붙여보면 API 형식이 전부 다르다.

- 인증 헤더가 다르고
- 요청 body 구조가 다르고
- 필수 옵션도 다르고
- 에러 방식도 다르다

그래서 `llm_client.py`를 중심으로 **프로바이더별 차이를 한 군데서 흡수하는 구조**를 잡았다.

핵심은 모델 문자열을 `provider:model` 형태로 다루는 방식이다.

```py
"openai:gpt-4o"
"anthropic:claude-sonnet-4-20250514"
"google:gemini-2.0-flash"
"ollama:llama3.1"
```

이렇게 해두면 council 단계에서는
"어느 회사 API인가?"보다  
**"어떤 조합으로 토론시킬 것인가?"**에 더 집중할 수 있다.

이 변화가 좋았던 이유는 단순히 모델 개수가 늘어서가 아니다.  
OpenRouter 한 곳에 묶여 있던 의존성을 줄이고, **내가 가진 모델 접근 수단을 더 직접적으로 다룰 수 있게** 됐기 때문이다.

## 토론형 시스템은 병렬 처리 없이는 바로 답답해진다

이 구조는 단계마다 여러 모델을 동시에 불러야 한다.

그렇다면 순차 호출은 사실상 선택지가 아니었다.  
하나씩 부르면 Stage 1부터 지연이 너무 길어진다.

그래서 핵심 호출은 `asyncio.gather()` 기반 병렬 처리로 묶었다.

여기서 중요했던 건 속도 자체보다도,  
**토론형 서비스의 기다림을 견딜 수 있는 수준으로 줄이는 것**이었다.

또 하나의 포인트는 부분 실패 허용이다.

모델 네 개 중 하나가 죽었다고 전체를 멈춰버리면 서비스가 너무 약하다.  
그래서 가능한 한 **살아남은 결과로 계속 진행**하도록 잡았다.

이건 화려한 기능은 아니다.  
하지만 실제로 오래 만져볼 서비스라면 이런 회복력이 생각보다 중요하다.

## 합의 과정을 더 읽기 쉽게 만들고 싶었다

원본 구조를 보면서 가장 아쉬웠던 건, 결과가 흥미로운데도 **사용자가 그 흥미를 읽어내기 어렵다**는 점이었다.

예를 들어 이런 질문이 생긴다.

- 지금 이 답변은 모델들이 꽤 비슷하게 본 걸까?
- 아니면 겉보기엔 결론이 하나지만 내부적으로는 많이 갈렸던 걸까?

그래서 이 프로젝트에서는 합의 과정을 더 가시화하는 쪽으로 손봤다.

백엔드에는 다음 계산 로직이 추가됐다.

- `calculate_consensus_score()`
- `calculate_aggregate_rankings()`
- `analyze_ranking_divergence()`

프론트엔드에는 `ConsensusIndicator`를 붙였다.

이제 사용자는 최종 답변만 보는 게 아니라,

- 판정 일치율
- 만장일치에 가까운 항목
- 심하게 갈린 항목

까지 같이 볼 수 있다.

결국 다중 LLM 시스템에서 중요한 건 "모델이 여러 개다"가 아니라  
**그 차이를 사용자가 이해할 수 있게 보여주느냐**였다.

## 긴 대화는 결국 컨텍스트 관리 싸움이 된다

실제로 몇 번 써보면, 이 구조는 컨텍스트가 꽤 빨리 무거워진다.

질문 하나에 답변 네 개,
평가 텍스트 여러 개,
최종 합성까지 붙으니 당연하다.

그래서 `context_manager.py`에 압축 흐름을 넣었다.

- 30K 토큰 임계치 초과 시 압축
- 오래된 대화는 요약
- 최근 2개 Q&A는 원문 유지
- 압축 전후 토큰 수를 메타데이터로 노출

이 부분이 중요했던 이유는,  
단순히 "토큰을 줄였다"보다 **긴 대화에서 서비스가 무너지지 않게 만들었다**는 데 있었다.

특히 백엔드에서 `context_compacted` 이벤트를 보내고, 프론트엔드에서 압축 알림을 띄우는 흐름은  
사용자가 시스템 내부 상태를 어느 정도 이해할 수 있게 만들어준다.

이런 작은 투명성이 생각보다 오래 남는다.

## 제품처럼 쓰기 위한 UX도 같이 손봤다

이런 프로젝트는 겉으로 보기엔 "주변만 손본 것"처럼 보일 수도 있다.  
그런데 실제로 오래 만져보면, 바로 그 주변이 사용감을 꽤 크게 바꾼다.

이 프로젝트에는 아래 기능들이 추가됐다.

- 다크모드
- 답변 복사
- 마지막 답변 재생성
- 후속 질문
- 대화 삭제
- 더 나은 대화 제목 생성
- 한국어 UI 브랜딩

기술적으로 극적인 기능은 아니다.  
그래도 이런 기본기가 있어야, 실험적인 데모가 아니라 **계속 열어보게 되는 앱**에 가까워진다.

## 마무리

이 프로젝트에서 남은 건 의외로 단순했다.

- 남의 구조를 읽고 이해할 수 있는가
- 그 구조의 강점은 살리고 약점은 보완할 수 있는가
- 단순 fork에서 끝나지 않고 실제 사용자 관점의 개선으로 이어갈 수 있는가

내가 가져가고 싶은 포인트도 거기에 있다.  
핵심 아이디어 위에서 **멀티 프로바이더 직접 연동, 합의 가시화, Context 압축, UX 보강** 같은 개선을 더해, 재밌는 데모를 실제 사용 흐름에 더 가까운 쪽으로 밀어본 경험이다.
