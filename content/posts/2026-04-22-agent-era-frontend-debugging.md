> 버그를 고치는 시간보다, 어디서부터 어긋났는지 찾는 시간이 더 길어졌다.

에이전트한테 "이 저장 버튼 안 된다, 좀 고쳐줘"라고 던지면 보통 뭔가는 바뀐다.

버튼은 눌린다.
로딩도 돈다.
겉으로 보면 해결된 것 같다.

그런데 새로고침하면 값이 다시 예전 상태로 돌아온다.
브라우저 콘솔에는 hydration 오류가 잠깐 지나갔고, 서버 로그에는 저장 성공이 찍혀 있다.
이럴 때부터 시간이 녹는다.

예전 프론트엔드 디버깅은 범위가 비교적 좁았다.

- 내 코드
- 브라우저 콘솔
- 네트워크 탭
- React DevTools

여기서 끝나는 경우가 많았다.

지금은 한 층이 더 끼어 있다.

- 에이전트가 만든 코드 조각
- 에이전트가 어떤 도구를 어떤 순서로 호출했는지
- 브라우저에서만 보이는 오류
- 서버 함수가 실제로 어떤 입력으로 실행됐는지
- 모델이 보고 있던 맥락이 무엇이었는지

그래서 요즘 디버깅은 "어느 줄이 틀렸나"보다 "어느 단계에서 상태가 처음 갈라졌나"를 찾는 쪽에 더 가깝다.

## 왜 지금 더 체감되나

이걸 괜한 과장이라고 보긴 어렵다.

2026년 2월 12일, Next.js 팀은 `Building Next.js for an agentic future`에서 에이전트가 브라우저를 못 본다는 문제를 정면으로 적었다.
거기서 나온 대응이 browser log forwarding과 Server Action invocation logging이다.

2026년 3월 18일 공개된 Next.js 16.2 AI 개선 항목에도 browser log forwarding과 agent devtools가 같이 들어갔다.
프레임워크 쪽이 이미 "브라우저, 서버, 에이전트가 같은 장면을 봐야 디버깅이 된다"는 쪽으로 움직이고 있다는 뜻이다.

이 흐름은 한 벤더만의 얘기도 아니다.
OpenAI 문서도 Responses API를 새 프로젝트 기본 선택지로 두고, Agents SDK는 tools, handoff, partial results, full trace를 전면에 놓는다.
OpenTelemetry GenAI semantic conventions도 아직 `Development` 상태이긴 하지만 model spans, agent spans, tool spans를 따로 보자고 적고 있다.

결국 질문이 바뀌었다.

"버그가 어디 있지?"보다
"이 실행 경로에서 상태가 어디서 갈라졌지?"가 먼저 나온다.

## 에이전트가 끼면 재현이 왜 더 어려워지나

에이전트 기반 개발에서 답답한 지점은, 같은 요청을 다시 던져도 꼭 같은 산출물이 나오지 않는다는 데 있다.

프롬프트가 같아도 코드가 조금 다를 수 있다.
도구 호출 순서가 바뀔 수 있다.
실패 후 재시도 과정에서 파일 상태가 달라질 수도 있다.

이러면 "같은 버그를 다시 만든다"는 말 자체가 흐려진다.

예전에는 브랜치만 같으면 비교적 비슷한 재현이 가능했다.
지금은 여기에 아래 조건이 더 붙는다.

- 어떤 프롬프트였는지
- 어떤 컨텍스트 파일을 읽었는지
- 어떤 도구가 먼저 실행됐는지
- 그 순간 워크스페이스 상태가 어땠는지

그러니 재현의 단위도 달라져야 한다.
이제는 결과 스냅샷만 붙잡고 있으면 부족하다.
실행 경로를 같이 남겨야 한다.

## 프론트엔드에서 특히 자주 꼬이는 세 군데

### 1. hydration mismatch

이건 예전에도 있었지만, 에이전트가 끼면 더 자주 마주친다.

모델이 빠르게 UI를 맞추려고 `Date.now()`, `Math.random()`, `window` 분기 같은 걸 아무 생각 없이 섞어 넣으면 서버와 클라이언트의 첫 렌더 결과가 금방 갈라진다.

React 19가 hydration mismatch를 예전보다 훨씬 읽기 좋은 단일 메시지와 diff 형태로 보여주긴 한다.
그래도 메시지가 좋아졌다고 원인이 자동으로 줄어드는 건 아니다.
생성된 코드가 왜 그런 분기를 만들었는지까지 봐야 끝난다.

### 2. client/server state drift

겉보기에는 더 짜증나는 문제다.

버튼은 성공처럼 보인다.
optimistic update도 들어간다.
토스트도 뜬다.

그런데 실제 서버 상태는 아직 예전 값이거나, 반대로 서버 저장은 끝났는데 클라이언트 캐시가 갱신되지 않는다.

에이전트가 코드를 만질 때 이 문제가 잘 생기는 이유는 간단하다.
한 파일에서는 UI 상태를 손봤는데, 다른 파일의 invalidation이나 revalidation 경로는 건드리지 않은 채 넘어가기 쉽기 때문이다.

특히 Server Action, client cache, optimistic UI가 같이 묶인 화면에서는 더 그렇다.
UI만 보면 맞아 보이는데 실제 상태는 벌써 갈라져 있다.

### 3. 브라우저에서만 터지는 race condition

에이전트는 대체로 "잘 돌아가게 보이는" 방향으로 빨리 손을 댄다.
문제는 그 과정에서 debounce, retry, transition, loading guard 같은 걸 겹쳐 넣는 일이 자주 생긴다는 점이다.

처음 한두 번은 멀쩡하다.
탭을 빠르게 옮기거나, 입력을 연속으로 넣거나, 느린 네트워크에서 다시 해보면 순서가 꼬인다.

이건 서버 로그만 봐서는 잘 안 잡힌다.
브라우저 이벤트 타이밍과 네트워크 응답 순서를 같이 봐야 한다.

## 브라우저 로그와 서버 로그가 따로 놀면 답이 안 나온다

요즘 프론트엔드 디버깅에서 제일 아쉬운 건 로그가 너무 흩어져 있다는 점이다.

브라우저 콘솔은 브라우저 콘솔대로 있고,
서버 함수 로그는 서버 로그대로 있고,
에이전트 도구 실행 내역은 또 다른 창에 있다.

이 상태에서는 각자 말은 하는데 서로 연결이 안 된다.

내가 먼저 남기고 싶은 최소 단위는 이런 정도다.

```ts
type DebugTrace = {
  traceId: string
  route: string
  buildId?: string
  agentRunId?: string
  toolCallId?: string
  serverAction?: string
  uiEvent?: string
}
```

핵심은 필드 이름이 아니다.
브라우저에서 난 일, 서버에서 처리한 일, 에이전트가 호출한 도구를 하나의 `traceId`로 묶을 수 있느냐가 더 중요하다.

그래야 이런 질문이 가능해진다.

- 이 hydration mismatch가 어느 코드 생성 이후부터 생겼지?
- 이 클릭 이벤트와 이 Server Action이 정말 같은 요청 흐름인가?
- 이 optimistic update가 서버 성공 전에 먼저 반영된 건가?
- 에이전트가 고쳤다는 파일과 실제 에러가 난 라우트가 이어져 있나?

디버깅은 결국 연결 문제다.
로그가 흩어져 있으면 버그도 흩어진다.

## 이제 기본 습관도 조금 바뀐다

예전 습관으로는 "에러 메시지 보고 해당 파일 열기"가 자연스러웠다.
지금은 그 전에 한 단계가 더 필요하다.

### 결과보다 경로를 남긴다

성공 여부만 남기면 나중에 쓸모가 없다.
어떤 입력으로 어떤 도구가 호출됐고, 어떤 서버 함수가 실행됐고, 어떤 브라우저 상태에서 실패했는지를 남겨야 한다.

### 생성된 코드를 한 덩어리로 믿지 않는다

"AI가 짠 코드라서 불안하다"는 얘기를 하려는 게 아니다.
문제는 출처보다 경계다.

한 번에 너무 넓게 바뀐 코드는 추적이 어렵다.
컴포넌트 분리, 상태 변경, 데이터 패칭 전략 변경, 로딩 UX 수정이 한 묶음으로 들어오면 나중에 어디서 갈라졌는지 찾기 힘들다.

그래서 생성 결과도 결국 추적 가능한 단위로 쪼개는 편이 낫다.

### observability를 나중 일로 미루지 않는다

표준은 아직 굳는 중이다.
그래도 방향은 꽤 분명하다.

모델 호출만 찍는 로그로는 부족하다.
agent span, tool span, browser event, server action이 한 흐름으로 묶여야 한다.
이걸 초반부터 안 남겨두면, 나중에는 버그를 고치는 시간보다 버그 설명을 다시 만드는 시간이 더 길어진다.

## 마무리

프론트엔드 디버깅은 여전히 콘솔에서 시작한다.
다만 이제 콘솔에서 끝나지 않는다.

에이전트 시대에 봐야 하는 건 에러 메시지 하나가 아니다.
브라우저와 서버와 도구 호출 사이에서 상태가 처음 갈라진 지점이다.

그래서 지금 디버깅의 핵심은 "AI를 믿지 말자"가 아니라 이쪽에 더 가깝다.
"추측하지 말고, 경로를 남겨라!"
